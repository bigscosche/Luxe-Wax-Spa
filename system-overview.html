System Overview

Architecture · Reuse · Ownership

What this site is

A static, zero-build, single-repo marketing site for a local service business. Pure HTML/CSS/JS, deployed to Netlify. No frameworks, no bundler, no database on the frontend. It's designed to be forked and rebranded for another local business in under a day.

Stack

Frontend

HTML + CSS in <style> blocks + vanilla JS. Google Fonts (Cormorant Garamond + Syne). No framework.

Host

Netlify. Drag-and-drop deploy or GitHub auto-deploy on push.

Booking

Vagaro. External link — no auth, no state in this site.

Lead capture

Contact form → Netlify Function → Supabase row + Slack webhook.

SEO

Schema.org JSON-LD (BeautySalon + LocalBusiness), sitemap.xml, robots.txt, city-level landing pages.

Analytics

Not yet wired. Netlify Analytics or Plausible recommended — add a single <script> to <head> of every page.

Reuse model

For a new business (cleaning company, contractor, another salon):

  1. Duplicate the repo.
  2. Change the palette + fonts in /assets/css/theme.css and in each page's inline <style> :root block.
  3. Find/replace the {{PLACEHOLDERS}} documented in /components/README.md.
  4. Swap services, service areas, and hero copy.
  5. Update sitemap.xml, JSON-LD addresses/geo, and canonical URLs.
  6. Point Netlify to the new repo.

Why inline per-page CSS

Every page ships as a self-contained file. This makes surgical edits safer, reduces moving parts, and lets the site be hosted anywhere — no build step, no framework lock-in. The trade-off is some duplication between pages; that's accepted in exchange for portability. The shared tokens in /assets/css/theme.css keep rebrands centralized.

What lives where

  • Public pages — root .html files + /service-areas/.
  • SEO — sitemap.xml, robots.txt, per-page JSON-LD in <head>.
  • Reusable markup — /components/ (copy-paste).
  • Design tokens / shared JS — /assets/.
  • Internal (noindex) — admin-notes, lead-structure, this page.

Deploy pipeline

  1. Push to GitHub main.
  2. Netlify build runs (no build command — it just publishes the repo root).
  3. Live at the Netlify subdomain (or custom domain when configured).
  4. Netlify Functions in /netlify/functions/ are auto-deployed with the site.
